This folder is the asset drop point for the Babylon renderer used by Rota
Estrategica.

## Board asset

`board.glb` is intended to be a single premium tabletop board model:

- thick dark wood/stone body
- aged brass outer frame
- reinforced corners
- visible side thickness
- inset 7x7 dark stone play area
- brass grid separators
- rivets and small surface details

The asset is generated by:

```powershell
blender --background --python tools/blender/create_route_board_glb.py
```

The Blender script automatically generates any missing texture sources. They can
also be regenerated explicitly with:

```powershell
python tools\assets\generate_route_board_textures.py
```

Optional preview render:

```powershell
blender --background --python tools/blender/create_route_board_glb.py -- --preview
```

That command writes both:

- `board-preview.png` - cinematic 3/4 preview with the full board visible.
- `board-preview-top.png` - top/near-top preview for checking all 49 tiles and
  the full outer frame.

Committed preview PNGs are archived outside `public/` at
`docs/archive/route-previews/` (CLEAN-05). Runtime loads only the GLB assets;
running a `--preview` generator can recreate previews under
`public/models/route/` when needed.

## Coordinate and orientation contract

The generated GLB is exported as Y-up glTF:

- board centered around world origin
- X axis = columns
- Z axis = rows/depth
- Y axis = vertical height
- playable surface faces upward
- row 0 is the back/top side when viewed from the intended 3/4 game camera

## Materials and texture sources

The Blender generator creates named materials backed by deterministic PNG source
maps under `public/models/route/textures/`:

- `route_tile_slate_a.png`
- `route_tile_slate_b.png`
- `route_tile_slate_c.png`
- `route_wood_painted.png`
- `route_bronze_brushed.png`
- `route_gem_blue.png`

Those maps are generated by `tools/assets/generate_route_board_textures.py` and
applied by `tools/blender/create_route_board_glb.py`:

- `DarkWood` / `WoodGrainDark` ‚Äî warm dark-wood body + grain inlay band
- `DarkStone` ‚Äî recessed sage/teal slate play bed
- `TileVariationA` / `TileVariationB` / `TileVariationC` ‚Äî sage/teal slate tiles
  (brighter, cleaner, toy-like; readable under the game pieces)
- `AgedBrass` / `BronzeDark` / `BrassPolished` ‚Äî clean warm brass/gold frame,
  bronze two-tone and polished gold highlights
- `RuneInlay` ‚Äî subtle brass rune engravings on selected tiles
- `CornerGem` ‚Äî signature faceted blue gemstone set into the four corner caps
  (glossy dielectric with a soft cyan glow; not overexposed)
- `WarmRivet` ‚Äî warm brass domed rivets
- `DeepShadow` ‚Äî recessed seams / contact shadow

The four corners are octagonal brass caps each set with a faceted blue gemstone
(`Corner_Gem_*`), matching the board concept sheet direction; the 7x7 grid, tile
centres and surface height are unchanged.

## Wall / obstacle asset (`wall.glb`)

`wall.glb` is a single premium tabletop **obstacle block** used by the Babylon
renderer for every wall cell on the Rota board ‚Äî it replaced the procedural
wall placeholders. It remains a self-contained asset that can also be previewed
in isolation.

It is built from these named objects:

- `Wall_Base` ‚Äî wide stone plinth so the block sits naturally on its tile.
- `Wall_Seam` ‚Äî thin dark recess between plinth and body (a seam shadow).
- `Wall_Body` ‚Äî tapered (frustum) dark-stone main body.
- `Wall_Trim_Lower` / `Wall_Trim_Upper` ‚Äî aged-brass straps.
- `Wall_Top_Cap` ‚Äî raised aged-brass cap.
- `Wall_Cap_Crown` ‚Äî subtle bright crown highlight on the cap.
- `Wall_Rivet_Front` / `_Back` / `_Left` / `_Right` ‚Äî warm brass bolt heads.

### How it was generated / how to regenerate

```powershell
& "C:\Program Files\Blender Foundation\Blender 5.1\blender.exe" --background --python tools\blender\create_route_wall_glb.py
```

Optional preview render (also writes the two PNGs):

```powershell
& "C:\Program Files\Blender Foundation\Blender 5.1\blender.exe" --background --python tools\blender\create_route_wall_glb.py -- --preview
```

That writes:

- `wall.glb` ‚Äî the exported obstacle (GLB only contains the wall meshes; the
  preview cameras/lights are added **after** export).
- `wall-preview.png` ‚Äî 3/4 preview.
- `wall-preview-top.png` ‚Äî top/near-top preview.

> Note: the generator looks the Principled BSDF up by node **type** and sets
> inputs by their **identifier**, not by display name, so it works on a
> non-English (e.g. Portuguese) Blender where the node/socket labels are
> localised. Setting colours by localised name silently leaves every material
> the default grey.

### Coordinate / orientation contract

Exported as Y-up glTF, matching `board.glb` and the Babylon grid:

- centered on the world origin in X/Z
- bottom rests on `Y = 0` (sits on the tile surface)
- `Y` is vertical height
- X = width, Z = depth

### Footprint and height

- footprint ‚âà `0.84 √ó 0.84` units (the 1.0-unit Babylon tile, with margin so
  it never covers neighbouring tiles)
- height ‚âà `0.60` units (tall enough to read as an obstacle, short enough not
  to hide the player/guardian pieces)
- file size ‚âà 88 KB (lightweight; safe to clone on every wall cell)

### Materials (procedural, no external textures)

- `WallDarkStone` ‚Äî dark stone body
- `WallSideStone` ‚Äî slightly lighter stone plinth
- `AgedBrass` ‚Äî caps and trim straps (metallic, not too bright)
- `DeepShadow` ‚Äî recessed seam
- `WallEdgeHighlight` ‚Äî subtle bright crown rim
- `WarmRivet` ‚Äî warm brass bolt heads

### Babylon usage (integrated)

`src/games/escape-maze/routeBabylonScene.ts` imports `wall.glb` once into a
disabled prototype (`loadWallAssetOnce`) and **clones** that prototype onto each
wall cell in `renderWalls` ‚Äî centered on the tile, base resting on the stone
surface (`BOARD_SURFACE_Y`), at scale 1 so it fits inside a single tile and
stays short enough not to hide the player/guardian/portal. Clones share the
GLB's geometry and materials and are non-pickable, so they never intercept tile
taps. If the import fails the renderer silently falls back to the procedural
wall blocks. Gameplay state stays in React/`useEscapeMaze`; the GLB is purely
visual.

### Prop coordinate / bounds contract

All prop GLBs follow the same contract as the board, wall and characters:

- centered on the world origin in X/Z
- bottom rests on `Y = 0`
- footprint stays inside one 1.0-unit tile
- `Y` is vertical height, `X` = width, `Z` = depth
- no external texture dependencies
- named meshes and named materials

### Babylon usage (props)

`src/games/escape-maze/routeBabylonScene.ts` imports each prop GLB once into a
disabled prototype and clones it onto the relevant board cells:

- `portal.glb` for the exit cell
- `light.glb` for uncollected collectible lights
- `trap.glb` for untriggered traps
- `shield.glb` for the available shield pickup

All cloned prop meshes are non-pickable, so invisible tile hitboxes remain the
only movement/input target. If any prop import fails, the renderer keeps the
procedural fallback for that prop. Triggered traps intentionally keep the darker
procedural fallback so the spent/triggered state stays visually distinct without
mutating shared GLB materials.

Do not encode gameplay state into these GLBs.
